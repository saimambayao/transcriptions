## Prototyping Fundamentals

### Purpose of Prototyping

Prototyping encourages testing functions and solutions in reality, with the desire to learn from users how to improve an offer on an ongoing basis. A prototype brings an idea into a form that allows potential users to experience and evaluate it.

### Key Questions When Building Prototypes

1. What are the basic functions for the user?
2. What hasn't been considered at all yet?
3. How has nobody ever done it before?

### Core Insight

The focus of prototyping is always on **learning**. Prototyping is possible at any time in the design thinking process. The added value of individual functions, a new product, or the result of a customer interaction can be tested with a prototype.

### First Prototype Characteristics

- Made using the simplest available materials
- Built as quickly as possible
- The simpler, faster, and cheaper, the less it hurts to reject it

**Available Materials**: Cardboard, paper, plastic cups, string, tape, aluminum foil, Lego bricks, and other readily available items.

## The Three Principles

### Principle 1: Love It, Change It, Leave It

This is the motto for all prototypes. After testing, you have three options:
- **Love it**: Keep what works
- **Change it**: Modify based on feedback
- **Leave it**: Discard what doesn't work

### Principle 2: Never Fall in Love with Your Prototype

**Warning**: Developing emotional ties to prototypes blocks objective evaluation.

- Be willing to make radical changes
- Schedule plenty of time for prototyping and testing
- Drive the prototype only as far as necessary to attain set goals

### Principle 3: Iterate, Iterate, Iterate

Prototyping is a never-ending story of continuous improvement.

**Process**:
1. Build prototype
2. Test with users
3. Gather feedback
4. Iterate and improve
5. Repeat

## Prototype Types and Resolution

### Resolution Levels

| Level | Phase | Purpose |
|-------|-------|---------|
| **Low** | Early phase | Quick exploration, basic concept |
| **Medium** | First approaches | Solution validation |
| **High** | End solutions | Final implementation |

### Complete Prototype Type Catalog

#### Physical Prototypes

| Type | Description | Resolution | Use Cases |
|------|-------------|------------|-----------|
| **Sketch** | Paper or digital, sketched or scribbled | Low | Practically everything |
| **Paper** | Building objects with paper/cardboard | Low | Products, furniture, accessories |
| **Physical Model** | 3D representation (3D printing, Lego) | Low | Products, spaces |
| **Mock-up** | Overall impression without functionality | Medium | Physical/digital products |

#### Digital Prototypes

| Type | Description | Resolution | Use Cases |
|------|-------------|------------|-----------|
| **Wire frame** | Early conceptual design showing arrangement | Low | Web sites |
| **Video** | Recording complex scenarios | Low/Medium | Experiences |
| **Photo** | Photo montage using editing software | Low | Products, experiences |
| **Open Hardware** | Analog/digital interfaces with motors/sensors | Medium/High | Electromechanical systems |

#### Experience Prototypes

| Type | Description | Resolution | Use Cases |
|------|-------------|------------|-----------|
| **Storyboards** | End-to-end customer journey in images | Low/Medium | Experiences |
| **Storytelling** | Communication of sequences and stories | Low/Medium/High | Experiences |
| **Role Playing** | Acting out customer experiences | Low/Medium | Experiences |
| **Bodystorming** | Physical acting of specific situations | Low | Physical experiences |

#### Business/Service Prototypes

| Type | Description | Resolution | Use Cases |
|------|-------------|------------|-----------|
| **Service Blueprinting** | Structured service description | Low/Medium/High | Services |
| **Business Model** | Business model canvas or lean canvas | Low/Medium/High | Business models |
| **Chart** | Showing correlations and changes over time | Medium/High | Processes, structures |

#### Lean Startup Prototypes

| Type | Description | Resolution | Use Cases |
|------|-------------|------------|-----------|
| **MVP** | Minimum Viable Product - only necessary functions | Low/Medium/High | Digital products, software |
| **MVE** | Minimum Viable Ecosystem - partner collaboration | Medium/High | Platforms, blockchain |
| **Fake Door** | Fake access to non-existent product | Low/Medium | Zynga, Dollar Shave Club |
| **Pretend to Own** | Pretending ownership (renting/leasing first) | Low/Medium/High | Zappos, Tesla |
| **Relabel** | Another product with own brand/packaging | Low | Products, services |
| **Pinocchio** | Rudimentary, non-functioning version | Low | PalmPilot |
| **Wizard of Oz** | User interacts with simulated system | Low/Medium | Speech-to-text experiments |

## Prototype Progression

### The Convergent Phase Path

```
Vision Prototype → Functional Prototype → Finished Prototype → Final Prototype
      ↓                    ↓                    ↓                    ↓
   Combine            Test critical          Integrate           Elegant
   all ideas          variables MVP          functions           realization
```

### A. Functional Prototype (MVP)

**Focus**: Critical variables only.

**Characteristics**:
- Concentrates on critical variables
- Tests intensively with potential users
- Creates critical functions for critical experiences
- Ensures minimum functionality for real-world testing
- Not all functionalities integrated at onset

**Purpose**: Serves as foundation to build upon.

### B. Finished Prototype

**Focus**: Multiple functions integrated.

**Characteristics**:
- Crucial for real user interaction
- "Only reality yields truth"
- Requires enough time for building
- Respective functionalities fully integrated

**Purpose**: Validates integrated solution.

### C. Final Prototype

**Focus**: Elegant realization.

**Characteristics**:
- Excels by elegance of thoughts and realization
- Convincing with simple functionality
- Obtains support from suppliers and partners
- Uses standard components to reduce costs

**Insight**: Prototypes convincing with simple functionality are usually successful when launched on the market.

### D. Implementation Plan

**Focus**: "How to bring it home."

**Considerations**:
- Identify who might put obstacles in the way
- Turn those affected into people involved
- Create win-win situations for all parties
- Not just product quality, but implementation quality matters

## Vision Prototype

The vision prototype is created at the "groan zone" - the transition from divergent to convergent phase.

### Components to Combine

1. Prior knowledge (use with caution)
2. Best initial ideas
3. Most important critical functionalities
4. New ideas from other industries and experiences
5. Initial user experience
6. Intriguing insights (e.g., from dark horse)
7. The simplest possible solution

### Process

**Time**: ~2 hours (depending on complexity)
**Testing**: At least 3 potential users
**Documentation**: Capture feedback in detail

### Key Questions for Vision Validation

| Question | Purpose |
|----------|---------|
| Does the vision generate enough attention that users absolutely want this solution? | Test desire |
| Does the vision give sufficient leeway for user's dreams? | Test flexibility |
| Is the value offer convincing? | Test value proposition |
| What else would users wish for to make the experience perfect? | Identify gaps |

## Boxing and Shelfing

Two methods for making prototypes tangible and discussable.

### Boxing Principle

Create a physical box (like a cereal box) that summarizes benefits and characteristics.

**Box Sides**:

| Side | Content |
|------|---------|
| **Front** | Product name, image, slogan, 2-3 promises |
| **Back** | Details about features, application, content |
| **Side 1** | WHO is the target customer/user? |
| **Side 2** | WHICH goals achieved? Problems solved? |
| **Side 3** | WHEN available? How to get it? |
| **Side 4** | WHERE used? Under what circumstances? |
| **Side 5** | WHY should user use it? |

**Box Variations**:
- Product box
- Problem box
- Solution box
- Project box (per project or work package)
- Process box (per process step)

### Shelfing Principle

Sort offerings into three portfolios:

**1. Solution Portfolio**
- Complete solutions offered
- Example: "Kitchens in country-style, urban for singles and families"

**2. Product Portfolio**
- Individual products
- Example: "Cupboards, table tops, electric appliances"

**3. Service Portfolio**
- Services offered
- Example: "Planning, delivery, assembly"

**Benefits**:
- Quickly detect gaps in offering portfolio
- Identify synergies
- Compare attributes (attractiveness, novelty, strategy contribution)
- Promotes discussion and collaboration

## Prototyping Workshop

A 7-step process for structured prototyping.

### Step 1: Test Variants on Team

Identify critical functions that users need.

### Step 2: Select Variant

Team decides which variant to build.

### Step 3: Build

Build one or multiple prototypes.

**Essential**: Provide enough material for building.

### Step 4: Feedback

Use "Green/Red" feedback:
- **Green**: "What I like about the prototype is..."
- **Red**: "I wish that the prototype..."

**Purpose**: Maintains positive mood while cultivating improvements.

### Step 5: Adapt Prototype

Improve prototypes based on initial feedback.

**Focus**: Concentrate on essential features and solutions.

### Step 6: Test with Users

Go out in pairs:
- One team member asks questions
- One team member observes

After returning, all team members document and share findings.

### Step 7: Adapt and Iterate

Based on findings:
- Improve prototypes
- Discard variants that don't work
- If none work, obtain more facts and customer needs

## Testing Prototypes

### The 4-Step Test Sequence

```
1. Test Preparation → 2. Conduct Test → 3. Document Results → 4. Infer Learnings
```

### Step 1: Test Preparation

**Define Clear Goals**:
- What do we want to learn?
- What do we want to test?
- With whom? Where?

**Create Question Map**:
- Formulate simple, clear, open questions
- Focus on core insights needed
- Let users talk about their experience
- Ask follow-ups: "Tell us what you think while you do that"

**Determine Test Scenario**:
- Describe exact sequence
- Provide necessary context
- Explain as simply as possible
- Let users experience, don't explain too much

### Step 2: Conduct Test

**Best Practice**: Test multiple ideas or variants.

**Why**:
- Single solution gets vague responses
- Multiple variants enable comparison
- More differentiated feedback

**Team Roles**:

| Role | Responsibility |
|------|----------------|
| **Moderator** | Help user cross to prototype situation, explain context, pose questions |
| **Actor** | Take on roles in scenario to create experience |
| **Observer** | Watch everything user does, film for later review |

**Motto**: "Never go hunting alone."

### Step 3: Document Results

**Active Observation**:
- Watch how users use (and misuse!) the prototype
- Don't immediately correct
- Use photos/video (with permission)

**Probing Questions**:
- "Can you say more about how it feels to you?"
- "Why?"
- "Show us why this would (not) work for you"
- Answer questions with questions: "What do you think this button is for?"

**Warning**: Resist marketing/voice-of-customer surveys!

### Step 4: Infer Learnings

- Improve prototypes
- Adapt persona based on insights
- Constant iteration contributes to learning
- "Fail fast - fail often" is key to realizing market opportunities

## Feedback Capture Grid

A 4-quadrant tool for systematic feedback documentation.

```
┌─────────────────────────┬─────────────────────────┐
│    + LIKES              │    Δ WISHES             │
│                         │                         │
│ Things liked or         │ Constructive criticism  │
│ worth mentioning        │                         │
│                         │                         │
├─────────────────────────┼─────────────────────────┤
│    ? QUESTIONS          │    💡 IDEAS             │
│                         │                         │
│ Questions cropping up   │ Ideas generated during  │
│ during experience       │ experience/presentation │
│                         │                         │
└─────────────────────────┴─────────────────────────┘
```

**Alternative Quadrants**:
- "I like..."
- "I wish..."
- "What if..."
- "What is the benefit?"

**Receiving Feedback**:
- See it as a gift
- Express gratitude
- Listen without justifying
- Ask if something is unclear

## A/B Testing

### When to Use

Suitable for:
- Simple prototypes
- Landing pages
- Single element testing (value proposition, button, etc.)
- Titles and descriptions
- Text volume, style
- Promotion offers
- Form length

### Requirements for Valid Results

1. Both versions tested concurrently
2. Predefined time period
3. Clearly predefined success criteria
4. Only one variable changed at a time

### Process

**Early Stage**:
1. Test person experiences variant A
2. Collect likes and change wishes
3. Repeat with variant B
4. Compare

**Online Testing**:
1. Distribute page views between A and B
2. Measure conversion rates
3. Check statistical relevance with calculators
4. New version B available only to new visitors (avoid confusion)

### Possible Outcomes

1. Result favors A
2. Result favors B
3. No statistically relevant preference
4. Insights on combining best of both variants

## Digital Testing Tools

### Selection Criteria

| Criterion | Description |
|-----------|-------------|
| **A. Prototype Upload** | Various types supported (single, variants, sequence) |
| **B. Scenario Description** | Users can understand the situation |
| **C. Question Types** | Predefined and open questions supported |
| **D. Community Selection** | Internal + external respondents possible |

### Example Questions for Testing

1. Evaluate the prototype with 1-5 stars
2. What do you like about the prototype?
3. What would you change in the prototype?

### Community Composition

- Internal organization members
- External customers
- Experts (by expertise area)
- Non-experts (fresh viewpoint)

**Insight**: Technically accomplished participants boost quality, but non-experts offer fresh viewpoints that shouldn't be ignored.

## Key Learnings Summary

1. **Start with user need** and market trend
2. **Build on the question** of what is to be tested
3. **No intrinsic value** - only customer-ascribed value counts
4. **Test as early as possible** in the real world
5. **Use available materials** - don't over-engineer
6. **Time box prototyping** - more time doesn't yield more results
7. **Match objective and maturity** of prototype
8. **Schedule plenty of time** for prototyping and testing
9. **Involve implementers early** in the process
10. **Apply boxing and shelfing** for portfolio testing
